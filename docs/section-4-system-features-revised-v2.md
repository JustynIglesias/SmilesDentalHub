## 4. System Features

This section presents the major features and functional capabilities of the proposed Smiles Dental Hub web-based system. It explains how the system responds to user actions and how each authorized user role is supported in performing clinic operations accurately, efficiently, and securely.

### 4.1 Stimulus/Response

This subsection presents the stimulus-and-response interactions between the users and the web system. It describes the actions initiated by users and the corresponding responses generated by the system, showing how the system validates requests, applies role-based permissions, stores data, and presents the appropriate output.

### 4.2 Summary of Key Functional Features

In addition to the stimulus-and-response interactions shown above, the system provides the following major functional features:

1. **Role-based access control.** The system displays only the modules and actions allowed for each user role, ensuring that receptionists, associate dentists, and admins access only the functions relevant to their responsibilities.
2. **Patient registration and duplicate checking.** The system supports guided patient intake through multi-step forms and validates possible duplicate patient records before final submission.
3. **Patient record management.** Authorized users can search, sort, filter, paginate, view, and update patient records, while inactive and archived records are managed separately.
4. **Dental record documentation.** The system supports dental charting, periodontal and occlusion recording, prescriptions, notes, and chart legend usage for clinical documentation.
5. **Service record management.** The system stores patient service histories, including service items, quantity, pricing, discounts, and total amounts, with editing limits depending on the user role.
6. **Patient document handling.** Authorized users can upload, view, delete, and manage supporting patient documents, subject to accepted file-type restrictions and role-based permissions.
7. **Export and printing.** The system can prepare a patient-record export preview for printing or saving as a PDF copy.
8. **Authentication, onboarding, and account recovery.** The system supports secure login, password reset through email verification codes, first-login onboarding for newly created staff accounts, failed-login alert handling, and protected session management.
9. **Profile maintenance and email verification.** Staff users can update profile information, change passwords, and request email changes that require verification codes sent to the new email address.
10. **Administrative management.** The admin can manage staff accounts, monitor inactive and archived records, archive and retrieve supported entries, process CSV-based patient migration imports, and enforce access restrictions for protected modules and maintenance actions.

### 4.3 Test Coverage Analysis and Recommended Additional Tests

The revised system-features section aligns with the compressed test-case document and shows that the current test set already covers many core workflows, including login, password recovery, patient registration, patient record viewing, dental and service record updates, patient logs, procedures maintenance, archiving and retrieval, email verification, onboarding, logout, access restriction, and session expiration.

However, additional tests are still necessary to ensure that all stated features in this section are validated directly and not only implied by broader workflows. The following items should be included in the test documentation:

1. **Protected intake navigation during patient self-registration.** The system states that staff must enter a password before leaving the intake page while a patient is still entering data, so this interruption-control behavior should be tested directly.
2. **Patient document upload validation and deletion.** Since the system allows document upload, viewing, and deletion with file-type restrictions, tests should verify valid upload, invalid file rejection, and successful document deletion.
3. **Patient active/inactive status change with patient-log recording.** The patient-record module allows status switching, and the action should be confirmed to update the record correctly and appear in the patient logs.
4. **CSV migration import validation and import summary.** The admin import feature should be tested to confirm file validation, row processing, duplicate handling, and display of created, updated, and skipped results.
5. **Role-restricted direct access to protected modules or actions.** Beyond menu visibility, protected routes and restricted actions should be tested when a user attempts access through direct navigation or a disallowed action path.

These additions improve traceability between the declared system features and the formal test evidence prepared for documentation and defense.
